# Phase 5 — 부하 테스트 인프라 (시나리오 기반 스트레스 테스트)

## Executive Summary

README.md 로드맵상 Phase 5는 "부하 테스트 인프라 (시나리오 기반 스트레스 테스트)"로 명시되어 있으나, 현재 `DhNet_StressTest`는 시나리오 기반이 아니다. 핸드셰이크/로그인까지는 자동으로 수행되지만 그 이후에는 실제 게임 로직(로비/룸/채팅)을 전혀 거치지 않고, 의미 없는 `TestPacket`을 단순 반복 전송할 뿐이다. 서버 측 핸들러(`TestController.cpp`)는 응답을 보내지 않으므로 RTT 측정도 원천적으로 불가능하다. 즉 현재 "부하 테스트"는 일방향 fire-and-forget 패킷만 던질 뿐, 실제 시나리오도 측정도 아니다.

또한 직전 코드 리뷰(로깅 시스템 작업 완료 시점)에서 Important 등급으로 지적된 사항이 있다: spdlog `async_overflow_policy::block` 정책 하에서, 워커 스레드가 로그 큐에 밀리면 `Session::HandleError`/`IocpCore::Dispatch` 등 핫 경로의 `LOG_WARN`이 블로킹되어 IOCP 워커 전체가 정체될 수 있다는 위험이다. 사용자는 이 위험을 사전에 정책 변경으로 막기보다, **Phase 5에서 실제로 부하를 걸어보고 실측으로 검증**하기로 결정했다. 따라서 Phase 5는 단순히 "동시 접속자를 늘려본다"가 아니라, 그 결과를 관측하고 판정할 수 있는 인프라까지 포함해야 한다.

## Current State Analysis

- `DhNet_Client/DhNet_StressTest/main.cpp` — `ClientService` 풀을 만들어 동시 접속 세션을 생성하고, 일정 주기로 ~10%를 폐기/재생성(churn)하는 구조는 있음. 그러나:
  - 실제 전송 내용은 `TestPacket`(응답 없는 fire-and-forget) 뿐 — `Req_RoomEnter`/`Req_RoomChat`/`Req_LobbyChat` 등 실제 패킷 흐름을 전혀 사용하지 않음
  - 서버가 `TestPacket`에 응답하지 않으므로(`TestController.cpp`의 `HandleTestPacket`이 `Send`를 호출 안 함) RTT 측정이 원천적으로 불가능한 구조
  - 측정/리포트 기능 없음 — 성공/실패 카운트, 지연시간(latency), 에러율 등 어떤 지표도 수집하지 않음. 콘솔에 "서비스 재생성 완료" 로그만 찍음
  - 서버 상태와의 교차 검증 없음 — 이미 구현된 Admin REST API(`GET /players`, `/lobbies`, `/rooms`, `/health`)를 전혀 활용하지 않음
- `ServerSession::OnConnected()`(`DhNet_Client/DhNet_Client/ServerSession.cpp:94-95`)가 핸드셰이크 완료 시 자동으로 `testuser`/`testpass`로 로그인 요청을 보냄 — 이 부분은 재사용 가능
- `LoginController`/`RoomController`/`ClientLobbyController` 핸들러는 존재하지만 콘솔 출력 위주이며 메트릭 집계용으로 설계되지 않음
- Admin REST API(`DhNet_Web`)는 이미 동작 중 (`GET /health`, `/players`, `/lobbies`, `/rooms`, `POST /players/{id}/kick`) — 외부에서 서버 상태를 폴링하는 용도로 그대로 활용 가능
- 로깅 시스템(spdlog 비동기 로거, `block` 정책)이 막 도입 완료된 상태이며, 고부하 시 병목 가능성이 미검증 상태로 남아있음 ([[project_phase5_logger_overflow_decision]] 메모리 참고)
- 서버 설정(`ServerSetting.cpp`)의 `DhNet_MAX_SESSION_COUNT` 기본값 1000 — 부하 테스트 시 이 한도를 의식해야 함

## Proposed Future State

1. **시나리오 기반 부하 클라이언트** — 의미 없는 `TestPacket` 반복 대신, 실제 플레이어 행동을 모사하는 시나리오(로그인 → 로비 채팅 → 룸 입장 → 룸 채팅 → 룸 퇴장 → 재입장 루프)를 각 가상 세션이 독립적으로 수행
2. **클라이언트 측 메트릭 수집** — 요청-응답 왕복 지연시간(RTT), 패킷 성공/실패/타임아웃 카운트, 연결 성공률을 세션 단위로 집계해 주기적으로 콘솔/파일에 리포트
3. **서버 상태 교차 검증** — Admin REST API를 폴링하여 "클라이언트가 인지하는 연결 수"와 "서버가 보고하는 플레이어/룸/로비 수"가 일치하는지 자동 검증 (불일치 시 누수/버그 신호)
4. **로거 병목 검증 시나리오** — 의도적으로 연결 폭주(connection storm)·에러 유발(잘못된 로그인 반복 등) 시나리오를 구성해 `LOG_WARN`/`LOG_ERROR` 핫 경로를 집중적으로 자극하고, 이 구간에서 서버 응답 지연이 튀는지 측정 → spdlog `block` 정책이 실제로 병목인지 실측 판정
5. **결과 리포트** — 테스트 종료 시(혹은 주기적으로) 텍스트/CSV 형태의 요약 리포트 생성 — 연결 성공률, 평균/최대 RTT, 에러 카운트, 서버-클라이언트 상태 불일치 여부

## Implementation Phases

### Phase 5.1 — 시나리오 클라이언트 코어 (M)
`TestPacket` 의존을 제거하고, 실제 패킷 흐름(`Req_LobbyChat`/`Req_RoomEnter`/`Req_RoomChat`/`Req_RoomExit`)을 순회하는 시나리오 루프를 `DhNet_StressTest`에 구현. `TestController.h`의 1초 sleep 제거(또는 시나리오 미사용 경로이므로 핸들러 자체를 stress test 빌드에서 미등록).

### Phase 5.2 — 메트릭 수집 (M)
세션별 RTT/성공/실패/타임아웃을 집계하는 경량 메트릭 구조체 + 스레드세이프 집계기 추가. 패킷 송신 시각을 기록하고 대응 응답(Noti/Res) 수신 시 RTT 계산.

### Phase 5.3 — 리포팅 (S)
주기적(예: 10초) 콘솔 요약 출력 + 테스트 종료 시 CSV/텍스트 파일 덤프.

### Phase 5.4 — 서버 상태 교차 검증 (S~M)
`DhNet_Web` Admin REST API(`/players`, `/lobbies`, `/rooms`, `/health`)를 별도 스레드에서 주기적으로 호출(HTTP GET, C++ 측에서는 간단한 WinHTTP 혹은 기존 인프라 재사용)하여 서버 보고 수치와 클라이언트 추적 수치를 비교, 불일치 로그 출력.

### Phase 5.5 — 로거 병목 실측 시나리오 + 결론 (M)
- "연결 폭주" 프로파일(짧은 시간에 다수 연결 시도 → 에러 유발 → `LOG_WARN` 집중 발생) 구성
- 이 프로파일 실행 중 RTT/연결 성공률이 정상 프로파일과 비교해 비정상적으로 튀는지 측정
- 결론에 따라 spdlog `overflow_policy`를 `block` 유지 또는 `overrun_oldest`로 전환할지 의사결정, 필요 시 `DhUtil/Logger.cpp` 수정 (이번 Phase 범위에 포함하되, 측정 결과가 "문제 없음"이면 코드 변경 없이 결론만 문서화)

## Risk Assessment and Mitigation

- **테스트 클라이언트 자체가 병목이 될 위험** — 클라이언트 머신의 IOCP 디스패치 스레드 수, 소켓 한도 등이 서버보다 먼저 한계에 도달할 수 있음. 완화: 동시 연결 수를 점진적으로 늘리며(ramp-up) 클라이언트 측 CPU/스레드 상태를 함께 관찰
- **`DhNet_MAX_SESSION_COUNT`(기본 1000) 한도 초과** — 의도적으로 한도를 넘기는 테스트를 한다면 서버의 거부 처리(슬롯 부족 응답)가 정상 동작하는지도 같이 확인. 완화: 사전에 한도 인지하고 시나리오 설계에 반영
- **DB 워커 풀(`DhNet_DB_POOL_SIZE`=4)이 로그인 폭주 시 병목** — 다수 동시 로그인 시 DB 커넥션 풀 대기열이 길어질 수 있음. 완화: 로그인 폭주 시나리오에서 DB 풀 지연도 같이 관찰(향후 별도 이슈로 분리 가능)
- **측정 자체의 오버헤드가 결과를 왜곡** — RTT 측정/집계 로직이 너무 무거우면 측정 행위 자체가 부하가 됨. 완화: 락-프리 또는 최소 락으로 집계, 측정 주기를 합리적으로 설정
- **로거 병목 실측이 "재현 안 됨"으로 끝날 가능성** — 이 경우도 유효한 결과(현재 구성에서는 문제 없음)이므로, 어떤 부하 강도까지 시도했는지 명시적으로 기록해 "검증된 범위"를 분명히 할 것

## Success Metrics

- 시나리오 기반 부하 클라이언트가 최소 N개(예: 500+) 동시 가상 세션으로 로그인→로비→룸→채팅→퇴장 사이클을 크래시 없이 지속 수행
