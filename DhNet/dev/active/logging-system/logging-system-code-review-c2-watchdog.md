# Code Review: CRASH 매크로 워치독 스레드 추가 (C2)

**대상 파일**: `E:\MyProject\DhNet\DhUtil\Macro.h`
**리뷰 범위**: 이번 세션에서 추가된 워치독 detached thread (CRASH 매크로 내부)
**참고(읽기 전용, 리뷰 대상 아님)**: `DhUtil/DeadLockProfiler.cpp`, `DhUtil/Lock.cpp`, `DhUtil/Logger.cpp`, `DhUtil/Logger.h`

## Executive Summary

CRASH 매크로에 "500ms 후 무조건 크래시"하는 detached `std::thread` 워치독을 추가해, `LOG_CRITICAL`의 enqueue나 `LoggerShutdown()`(spdlog async_logger의 `flush()`, overflow policy `block`)이 무한 블로킹되더라도 fail-fast가 보장되도록 한 의도는 타당하고 실제 위험(DeadLockProfiler::PushLock이 자신의 mutex를 쥔 채 CRASH를 호출 → flush 블로킹 시 서버 전체 락 체인이 멈추는 시나리오)을 직접적으로 해소한다.

다만 구현에 한 가지 **Critical** 수준 결함이 있다: `std::thread`의 생성자/멤버 함수가 리소스 부족 시 `std::system_error`를 던질 수 있는데, CRASH 매크로 어디에도 예외 처리가 없다. CRASH는 정의상 "이미 비정상 상황"(데드락 감지, 락 타임아웃, assert 실패)에서 호출되므로 스레드 풀/핸들 리소스가 정상보다 고갈되어 있을 가능성이 평소보다 높은 시점이며, 이 경우 워치독 생성 자체가 예외를 던져 CRASH 호출부(`Lock::WriteLock` 등 noexcept를 기대하는 호출 체인일 수 있음)로 예외가 전파되어 의도와 다른 종료 경로(std::terminate 또는 호출자의 catch에 흡수되어 fail-fast 무력화)를 탈 수 있다. 이 외에는 설계가 견고하며, Important/Minor 수준의 개선 여지만 있다.

## Critical Issues

### C1. `std::thread` 생성 실패 시 예외가 처리되지 않음

```cpp
#define CRASH(cause)					\
{										\
	std::thread([] {					\
		...								\
	}).detach();						\
	LOG_CRITICAL(...);					\
	...
```

- `std::thread`의 생성자는 새 스레드를 시작시키지 못하면 `std::system_error`를 던진다(리소스 한도 초과, 핸들 고갈 등). CRASH는 보통 비정상 상황(데드락, 락 타임아웃, 메모리/리소스 문제로 인한 부수적 실패)에서 호출되므로, 시스템 리소스가 이미 압박받고 있을 가능성이 평소보다 크다.
- 이 예외는 CRASH 매크로 바깥, 즉 호출자의 스택으로 그대로 전파된다. 구체적으로:
  - `Lock::WriteLock`/`ReadLock`(`Lock.cpp:36, 89`)에서 호출되는 CRASH가 예외를 던지면, 락을 호출한 임의의 게임 로직 코드까지 예외가 전파된다. 그 코드가 `noexcept`이거나 IOCP 워커 스레드의 콜백처럼 예외를 적절히 처리하지 못하는 컨텍스트라면 `std::terminate()`가 즉시 호출되어 버린다(이 경우 결과적으로는 종료되긴 하지만 LOG_CRITICAL/flush가 전혀 실행되지 않아 원인 로그가 통째로 사라짐 — fail-fast의 "원인 기록" 목적이 무력화).
  - 반대로 예외가 try/catch에 흡수되는 호출 경로가 있다면, **크래시 자체가 일어나지 않고 넘어가 버려** fail-fast가 완전히 무력화된다. 이번 패치가 막으려던 시나리오(흐름이 멈춰서 영원히 안 죽는 것)보다 더 나쁜 시나리오(아예 안 죽고 조용히 넘어가는 것)가 새로 생긴다.
- 권장: 워치독 스레드 생성을 예외로부터 보호한다.
  ```cpp
  try {
      std::thread([] { ... }).detach();
  } catch (...) {
      // 워치독 생성 실패 자체도 fail-fast 대상.
      // 메인 경로(LOG_CRITICAL 이하)가 그대로 이어지므로 별도 처리 불필요할 수도 있으나,
      // 최소한 예외가 CRASH 호출자로 전파되지 않도록 흡수해야 한다.
  }
  ```
  메인 경로(LOG_CRITICAL → LoggerShutdown → null write)는 워치독 생성 여부와 무관하게 항상 실행되어야 하므로, 워치독 생성 실패는 "워치독 없이 메인 경로만 시도"로 격하시키는 것이 안전하다(워치독이 없으면 이번 패치 적용 전 상태로 돌아갈 뿐, 새로운 위험이 추가되지는 않는다).

## Important Improvements

### I1. 500ms 동안 워치독 스레드 생성/스케줄링 비용이 fail-fast 지연에 더해짐

`std::thread` 생성은 OS 스레드 핸들/스택 할당을 수반하므로 마이크로초~밀리초 단위 비용이 든다. 정상 케이스(메인 경로가 즉시 크래시하는 경우)에도 CRASH 호출마다 이 비용이 추가된다. CRASH는 핫 패스가 아니라 항상 비정상/예외 상황에서만 호출되므로 실질적 영향은 미미하지만, "워치독은 프로세스 종료와 함께 소멸되어 영향 없음"이라는 설계 의도 설명에 스레드 생성 비용 자체는 별개 항목으로 남는다는 점은 인지해 둘 필요가 있다. 실용적으로는 문제 삼을 수준이 아니므로 Minor에 가깝지만, 다중 CRASH가 짧은 시간에 폭주하는 시나리오(I2 참고)와 함께 고려할 가치는 있다.

### I2. 동일 프로세스 내 여러 스레드가 동시에 CRASH를 호출하는 경우

- 예: 여러 락에서 동시에 `LOCK_TIMEOUT`이 발생하거나, `DeadLockProfiler::DFS`가 사이클을 여러 개 동시에 잡는 등. 이 경우 워치독 스레드가 여러 개 뜨지만:
  - 각 워치독은 독립적으로 자신의 500ms를 기다린 후 *서로 다른* `crash`(스택/람다 지역 변수, 각 스레드 전용)에 null write를 시도한다. 모두 같은 프로세스 주소공간에서 AV를 일으키므로 결과적으로 동일한 효과(프로세스 종료)이며 경쟁 조건으로 인한 추가 위험은 없다.
  - 메인 경로 중 하나가 먼저 크래시하면 프로세스가 즉시 종료되고 나머지 워치독들은 OS에 의해 함께 정리된다 — 문제 없음.
  - 메인 경로와 워치독이 "동시에" AV를 일으키는 경우도 결과는 동일(프로세스 종료, WER 덤프 생성)하므로 실질적 위해는 없다. 단, 덤프가 어느 스레드의 크래시 지점을 기준으로 캡처되는지는 비결정적이 되는데, CRASH는 이미 `LOG_CRITICAL`로 원인(cause, 파일, 라인)을 로그에 남긴 후이므로 덤프의 크래시 스택이 어느 스레드든 진단에는 로그가 보완 정보를 제공한다. 실무적으로 큰 문제는 아니다.
- 결론: 동시 호출 시나리오 자체는 안전하다. 람다가 캡처하는 것이 없는 점(전역/정적 상태에 접근하지 않음)도 동시성 문제를 일으키지 않는 이유 중 하나로 적절하다.

### I3. `DeadLockProfiler::PushLock`/`PopLock`에서 CRASH 호출 시 `m_lock`을 쥔 채로 워치독이 발동하는 시점

- `DeadLockProfiler.cpp:48, 51, 119`에서 CRASH가 호출될 때 `std::lock_guard<std::mutex> guard(m_lock)`이 스코프에 걸려 있다. 워치독이 500ms 후 발동해 프로세스를 죽이는 것은 이 mutex 상태와 무관하게 항상 성공하므로(프로세스 강제종료이지 mutex 획득이 아님) 의도대로 동작한다. 다만 메인 경로가 멈추는 동안(`m_lock`을 쥔 채 LOG_CRITICAL enqueue/flush가 블로킹) 같은 프로세스의 다른 스레드들이 `PushLock`/`PopLock`에서 같은 `m_lock`을 기다리며 함께 멈추는 것은 패치 이전과 동일하게 여전히 발생한다 — 이번 패치는 "그 정지 상태가 영원히 지속되지 않고 500ms 후 종료된다"를 보장하는 것이지 "정지 자체를 막는 것"은 아니다. 의도된 범위이므로 결함은 아니나, 문서/주석에 "워치독은 정지를 끝내는 것이 목적이며, 정지 자체를 방지하지 않는다"는 점을 한 줄 더 명확히 해두면 추후 오해를 줄일 수 있다.

## Minor Suggestions

### M1. 500ms 타임아웃 값의 근거

- `Lock.h`의 `ACQUIRE_TIMEOUT_TICK = 10000`(10초)과 비교하면 500ms는 충분히 짧아 "거의 즉시 죽는다"는 인상을 준다. 정상 케이스(메인 경로가 정상적으로 수십 ms 내 끝나는 경우)와 비정상 케이스(영원히 블로킹) 사이의 간극이 크므로 500ms는 합리적인 절충값으로 보인다.
- 다만 `flush_every(std::chrono::seconds(3))`(`Logger.cpp:28`)는 별도 타이머 기반 자동 flush이며 CRASH 경로의 `LoggerShutdown()`(동기 `flush()`)과는 독립적이므로 500ms와 직접 비교할 대상은 아니다. 혼동을 막기 위해 주석에 "이 값은 spdlog의 flush_every 주기와 무관하며, CRASH 경로 자체의 fail-fast 보장 시간이다" 정도를 덧붙이면 좋다.
- 대안으로 제시된 "volatile 포인터 사용"은 이 컨텍스트에서는 메인 경로의 null write(`*crash = ...`)가 최적화로 제거되는 것을 막기 위한 것이라면 M2와 같은 논의이고, 워치독의 타임아웃 메커니즘 자체에 대한 대안으로는 큰 의미가 없다(타이머 기반 접근 자체가 이미 단순하고 충분히 견고함). 500ms 값을 매크로 상단에 명명된 상수(`CRASH_WATCHDOG_TIMEOUT_MS` 등)로 추출하면 추후 조정이나 다른 코드에서의 재사용/참조가 쉬워진다.

### M2. 컴파일러가 null 포인터 쓰기를 최적화로 제거할 위험 — 워치독에도 동일하게 존재

- 메인 경로와 워치독 람다 모두 동일한 패턴(`unsigned __int32* crash = nullptr; __analysis_assume(crash != nullptr); *crash = 0xDEADBEEF;`)을 사용한다. `__analysis_assume`은 MSVC 정적 분석기(코드 분석/IntelliSense)에게 "이 지점에서 crash가 null이 아니라고 가정하라"고 알려주는 어노테이션으로, 코드 분석 경고(예: C6011 null 역참조 경고)를 억제하는 용도다. **런타임 최적화 동작에는 영향이 없다** — 즉 릴리스 빌드 옵티마이저가 "crash는 항상 nullptr이므로 이 대입은 정의되지 않은 동작(UB)이고 죽은 코드로 취급해 제거해도 된다"고 판단할 가능성을 막지는 못한다.
- 다만 실무적으로 `nullptr`에 대한 쓰기는 일반적인 옵티마이저가 "이 함수는 항상 UB를 일으키므로 호출 자체를 제거 가능"이라고 추론하기에는 매크로 인라인 컨텍스트(함수 경계가 없는 블록)와 람다 내부(즉시 실행되는 익명 함수, LTO/inlining 대상이 될 수 있음)에서 패턴이 다소 다르다. 워치독 람다는 별도 스레드 진입점이라 컴파일러가 "호출되지 않는 코드"로 판단해 통째로 제거할 근거가 메인 경로보다는 약하지만(스레드로 전달되는 콜러블이라 호출 여부를 정적으로 알 수 없음), 안전성을 더 확실히 하려면 두 곳 모두 `volatile` 포인터(`volatile unsigned __int32* crash = nullptr;`) 또는 `RaiseException`/`__fastfail`/`std::abort()` 같은 명시적이고 컴파일러가 제거할 수 없는 종료 수단으로 교체하는 것이 더 견고하다. 이는 이번 패치 이전부터 존재하던 기존 메인 경로의 위험을 워치독이 그대로 복제한 것이며, 워치독 추가로 인해 악화되지는 않았다.

### M3. 워치독 람다의 캡처 없음은 안전하지만, 디버깅 시 스레드 식별이 어려움

- 캡처가 없어 동시성 문제는 없지만, 크래시 덤프 분석 시 어느 CRASH 호출(`cause`, 파일/라인)이 워치독을 띄웠는지가 워치독 스레드의 콜스택만으로는 드러나지 않는다(메인 경로의 LOG_CRITICAL 로그를 함께 봐야 알 수 있음). 디버깅 편의를 위해 `cause`/`__FILE__`/`__LINE__`을 워치독 스레드 이름(`SetThreadDescription` 등, Windows API)에 반영하면 덤프 분석 시 워치독이 발동했는지 메인 경로가 발동했는지 즉시 구분 가능해진다. 필수는 아니다.

## Architecture Considerations

- **fail-fast 보장의 계층화**: 메인 경로(로그+flush 후 크래시) → 워치독(타임아웃 기반 무조건 크래시)이라는 2단 방어선은 합리적인 설계 패턴이다. 단, C1에서 지적한 예외 안전성 문제를 해결하지 않으면 이 2단 방어선의 "바깥쪽" 방어선(워치독) 자체가 예외로 인해 무력화되거나, 예외 전파로 인해 오히려 "안쪽" 방어선(메인 경로)까지 도달하지 못하는 역설적 상황이 생길 수 있다. C1 수정이 이 설계의 핵심 전제(워치독은 절대 실패하지 않는다)를 지키기 위한 필수 조건이다.
- **DeadLockProfiler/Lock과의 일관성**: 두 호출부 모두 CRASH를 동기 함수 호출처럼 사용하고 있어(반환을 기대하지 않음) 매크로의 새 워치독 추가가 기존 호출부의 동작 가정을 깨지 않는다. 워치독은 어디까지나 "보험"이며 정상 흐름에서는 한 번도 발동하지 않는 것이 기대값이므로, 기존 호출부 코드 변경은 필요 없다 — 이는 이미 잘 지켜지고 있다.
- **#include 추가의 영향**: `<thread>`, `<chrono>`를 `Macro.h` 상단에 명시적으로 추가한 것은 합리적이다. `Macro.h`는 `DhUtil`뿐 아니라 `DhNet_Server`, `ServerCore` 등 여러 프로젝트의 pch에 의존하지 않고 단독으로 포함될 수 있어야 하므로, 헤더 자체가 필요한 표준 헤더를 직접 포함하는 것은 의존성 누락을 방지하는 올바른 방향이다.
