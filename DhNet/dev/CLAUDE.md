# DhNet 작업 워크플로우 (비공개)

## 에이전트 실행 주의사항

`.claude/agents/`의 에이전트들(`code-architecture-reviewer` 등)은 `subagent_type`으로 직접 등록되어 있지 않다.
Agent 도구 호출 시 `subagent_type="general-purpose"`를 사용하고, 프롬프트 안에 해당 에이전트의 페르소나와 리뷰 체크리스트를 직접 포함시킨다.

## 작업 워크플로우 (자동 실행 규칙)

### RULE 0: 구현 전 계획 수립 + 사용자 컨펌 [필수]
어떤 코드 작업이든 **구현에 들어가기 전에** 반드시 계획을 먼저 세우고 사용자에게 컨펌을 받아야 한다.
- 계획 없이 바로 코드를 작성하지 않는다.
- 계획을 제시한 후 사용자가 명시적으로 승인("진행해줘", "맞아", "ok" 등)할 때까지 구현하지 않는다.
- **절대로 건너뛰지 않는다.**

### RULE 1: 새 작업 시작 시 → dev-docs 자동 실행 [필수]
사용자가 새로운 기능/작업 구현을 요청하면 **코드 작성 전에** 반드시 Skill 도구로 `dev-docs`를 실행한다.
- 트리거: "~구현해줘", "~만들어줘", "~추가해줘", "새 기능", "시작하자" 등
- 실행: `Skill("dev-docs", "[작업명]")`
- **절대로 건너뛰지 않는다.**

### RULE 2: 코드 작업 완료 후 → dev-docs-update + code-architecture-reviewer 자동 실행 [필수]
C++ (.cpp/.h/.hpp) 또는 C# (.cs) 파일을 하나 이상 작성/수정하고 나면 **응답 마지막에** 반드시 두 가지를 순서대로 실행한다.
1. `Skill("dev-docs-update")` — 작업 상태 저장
2. `Agent(subagent_type="general-purpose", ...)` — code-architecture-reviewer 페르소나를 프롬프트에 직접 포함해 코드 리뷰 실행
- **둘 다 반드시 실행. 어느 하나도 생략 불가.**

### RULE 3: 커밋은 절대 자동으로 하지 않는다 [필수]
`git commit`은 항상 사용자가 직접 한다. Claude는 커밋 메시지 추천까지만 하고 실제 커밋 명령은 실행하지 않는다.
- 사용자가 명시적으로 "커밋해줘"라고 요청한 경우에만 예외.
- 커밋 메시지에 `Co-Authored-By: Claude ...` 줄을 절대 추가하지 않는다 (amend 포함).
- **절대로 건너뛰지 않는다.**

### 순서
1. 새 기능 요청 → **계획 수립 및 사용자 컨펌** → **`dev-docs` 실행** → 코드 작성
2. 코드 작성
3. 코드 완성 → **즉시 `dev-docs-update` 실행** → **즉시 `code-architecture-reviewer` 에이전트 실행**
4. 커밋 메시지 추천 → **사용자가 직접 커밋**
